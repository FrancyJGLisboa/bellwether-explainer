US ag intelligence, as published — queryable, cited, and scored.

Bellwether stores U.S. and global farm-data reports exactly as they were published, and ties every number it hands back to the document and release date it came from. Later government revisions don't overwrite what was on record at the time.

Because the record is dated, the tool's own forecasts can be filed in advance and graded later against what actually happened — which is the only honest way to tell whether any of it is worth reading. The evidence is in § 05.

12 sources, one file per release, frozen on the day it published.

Bellwether is a Python tool that pulls in a fixed set of U.S. and global agricultural reports and stores each one frozen as it looked on its release day — later government revisions don't overwrite what was on record at the time.

That “as published” discipline is not a nicety. It is what makes an honest track record possible at all: a system that only serves today's revised figures leaks the answer backwards into the past, and every backtest built on one is fiction.

It keeps a written record — call it a ledger — of specific forecasts too: a number, for a named report, due on a named future release date. Those entries sit and wait, and get checked off as right or wrong once the real report lands. The ledger is append-only, and one call per series per release: a call that turns out wrong stays wrong, on the record, and cannot be quietly re-rolled.

If you already know the AI jargon: this is a RAG system — “retrieval-augmented generation,” where the AI answers from documents it looks up rather than from memory. Bellwether does the looking-up with full-text and structured search over the frozen archive (FTS and SQL), not the “meaning-similarity” embeddings most such tools use. It still handles meaning — ask for corn and it also searches maize — but every such jump is written down in a list you can read and diff, not inferred inside a model. The dividing line is deterministic vs learned, not keyword vs meaning: for dated, citeable records you want a retrieval step whose behaviour you can audit and test, and that reach is measured on a held-out slice rather than asserted.

A number with no date attached is a rumour.

Ask most tools a question about supply, weather risk, or trader positioning and you get a fluent, confident paragraph back. What you don't get is a way to know if any number in it is current, revised, or made up.

The report a number came from might have been updated since — quietly, in place — and you'd never know your answer was reading the old version. Or the number was never in any report at all; it was smoothed over from memory into something that merely sounds right. Either way, you're trusting it on vibes, because there's no date and no document attached to check against.

And even the confident calls — "stocks are tightening," "conditions are deteriorating" — almost never get followed up. Nobody circles back three months later and asks whether that call was right. Without that follow-up, a wrong call and a right one look identical, and bad analysis survives simply because it's never tested against what actually happened.

It won't make a forecast that can't be wrong.

The part that makes this more than a data cache: every forecast it produces has to be specific enough to fail.

Each view ends in one plain sentence — a number, for a named report, on a named future release date — written down before the outcome is known, not after.

And each call is graded against the archive's own climatology: how often this series has historically moved past its standing estimate over the same horizon. Being right when the answer was obvious earns nothing. Beating that baseline is the only thing that counts as skill.

A forecast that can't be wrong isn't a forecast — it's a horoscope.

Every figure carries its source file and release date — or it is vetoed.

An AI assistant reaches the archive over MCP (Model Context Protocol — the standard way a connected AI tool calls out to external data instead of guessing from memory). Every figure the assistant hands back must carry the exact document and release date it came from.

Before answering, it checks freshness first: a status call flags any source that's out of date, and that flag gets disclosed in the answer rather than hidden. Questions outside its lane — cash and basis prices, spreads and term structure, intraday quotes, trader-consensus surveys, county-level crop detail, weather forecasts — are refused by name instead of guessed at.

Directional claims like "stocks are tightening" are never eyeballed — a dedicated calculation (interpret_change) computes the direction and size of every move from the actual figures, and any sentence whose wording contradicts the computed sign is vetoed before it reaches you.

Beyond the citation, every figure carries a provenance stamp: the source's tier (issuer or aggregator), whether the number is exact or rounded, and whether that series revises in place. A reader can weigh a number, not just trace it — and a separate integrity scan verifies that every stored fact's cited source file actually resolves to a real document in the archive.

After each data refresh, a background job resolves any pending forecasts against the newly landed report and logs each one as a hit or a miss, alongside a running score.

13 of 23 calls right, where climatology expects seven.

Every AI tool claims it doesn't hallucinate. Almost none can tell you whether its analysis is any good. Grounding a number and being right about the future are different problems — and only the second one decides whether the thing is worth reading.

So bellwether was made to forecast, in the dark, and then graded.

Twenty-four questions: will the WASDE report three months from now print corn, soybean or wheat stocks and exports above or below where USDA has them standing today? Each one was answered by a fresh AI session that could reach only a copy of the archive holding data published on or before that question's date. Later data was physically absent from the only database it could open — not discouraged, absent. One of the twenty-four sessions died on an infrastructure error before answering; the twenty-three that completed are what's graded.

Every release that settles these questions landed after the model's training cutoff. It cannot have remembered the answers, because they did not exist when it was trained. That single property is what separates a measurement from a magic trick — and it is only available because the archive keeps every release exactly as published.

Thirteen calls right out of twenty-three — where the archive's own history says about seven should have been. The odds of managing that by luck are 7 in 1,000 (exact Poisson-binomial, each call scored against its own frozen baseline).

The naive desk beside it scores about zero, which is precisely what a null is supposed to do. That number is the control: when it drifts away from zero, the referee is broken and no result beside it can be believed.

It reads the fundamentals better than chance — and it can prove that on data it could not have memorised.

The direction is trustworthy. The confidence is not.

A tool that only lists its strengths is asking to be trusted on vibes — which is the exact thing this project exists to refuse.

The direction is trustworthy. The confidence is not. Ask how sure it is and it says about 70%. Measured, it is right about 57% of the time. It is not hallucinating — every figure behind that call traces to a real dated document — it simply over-trusts its own reasoning, the way a bright junior analyst does.

So take the direction, and do not size a decision on the percentage. That gap is a known defect with an open fix, not a footnote we hope you skip. A confident wrong number that passes every citation check is more dangerous than an obvious hallucination, precisely because nothing catches it.

The live ledger is young. Its forecasts are filed and pending; the earliest settle at the next monthly report. The evidence above comes from the backtest, which grades the same machinery against releases that have already landed.

It holds no prices at all — on purpose. Everything about price is out of corpus: no futures settlement, no cash or basis prices, no spreads or term structure, no intraday quotes, no weather forecasts, no trade-consensus surveys. Ask, and it refuses by name rather than improvise. That boundary is deliberate: the price legs belong to other tools, and a system that guesses at them forfeits the only thing that makes this one worth having.

Check a claim, file a call, or let it grade itself.

Three shapes of the same idea: don't trust a claim about ag markets you can't trace back to a dated source, and don't let a forecast go unchecked.

1
verification

Sanity-check a market claim before you repeat it

Someone says corn ending stocks are tightening. Instead of taking that on faith, query the archive: it pulls only the report series that claim actually needs, cites each figure to its release date, and says plainly if part of the question (like a price move) is outside what it covers.

2
prediction tracking

Write a forecast that can be proven wrong

Instead of analysis nobody checks later, log a call bound to a specific future report release. The same tooling replays it over past windows with no lookahead — the sandboxed backtest in eval/ is exactly how the numbers in the previous section were produced, and you can rerun it yourself.

3
automation

Let it run on a schedule and report its own scorecard

Wired to a recurring job, it refreshes the underlying data and grades any forecasts that just came due automatically — surfacing hits and misses without you having to check in and remember to look.

Clone it, sync, build the archive, ask.

No separate install step is listed for this repo beyond syncing dependencies — the CLI (bellwether) handles archive-building and querying from there.

uv sync
  1. Prerequisite A Python project managed with uv (the dependency/runner tool). Have uv installed; nothing else is required first.
  2. Install dependencies Run uv sync — pulls in the project's pinned packages, including claimcheck, pypdf, and xlrd.
  3. Build the local archive Run uv run bellwether all --since 2020 --root archive — fetches every source since 2020 into a local archive/ folder. Expect a real download; this populates the archive, it doesn't just configure something.
  4. Confirm it worked Run uv run bellwether status — prints each source with a freshness flag. A clean run shows the sources you just pulled as current, not stale.
  5. Query the archive directly Run uv run bellwether facts && uv run bellwether fts to see the stored, dated records themselves — this is what you now have: a local, citeable archive instead of a black-box summary.
  6. Grade it yourself Run uv run python eval/backtest.py --policy mech --grid v3 for the zero-token null desk, and read eval/backtest_v3_protocol.md — the protocol was committed before any answers existed, and the commit order is the audit.
  7. Next Keep it current with uv run bellwether refresh, or pull a single source on demand — e.g. uv run bellwether wasde --since 2020 or uv run bellwether nass --since 2000.
